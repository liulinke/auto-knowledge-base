# 自动化网络数据采集与本地知识库构建需求书

## 1. 项目概述

本项目旨在通过多次自动化的网络查询，围绕用户设定的特定主题和过滤条件，抓取、清洗并持久化存储网络数据，最终构建一个结构清晰、具备去重能力、可离线浏览的**本地知识库**。

系统采用 **deepagents + LangChain + LangGraph** 技术栈实现，后端支持**多用户隔离**：每个用户拥有独立的存储目录，每个知识库对应一个独立文件夹，用户可以通过多次查询**持续增量构建**同一个知识库。

## 2. 核心功能流程

1. **输入优化**：接收用户设定的主题和查询条件，由 AI 自动优化并扩展用户的原始输入。
2. **关键词生成**：基于优化后的主题，自动生成用于搜索引擎的精准查询关键字。
3. **网络搜索与下载**：调用搜索引擎或爬虫接口，自动下载搜索结果的前 $N$ 条相关数据（数量可配置）。
4. **增量更新与去重**：利用文件的元数据（Metadata）进行比对，在多次查询和下载过程中，自动剔除重复内容。
5. **AI 摘要生成**：每篇入库文章由 LLM 自动生成内容摘要（Summary），写入该文章的元数据文件，供 RAG 检索与人工浏览使用。
6. **索引重建**：每次增量更新后，自动重新生成知识库的 `README.md` 与离线浏览入口 `index.html`。

## 3. 多用户后端存储架构

后端存储按「用户 → 知识库」两级隔离：

```
<data_root>/
└── <user_id>/                  # 每个用户一个独立目录
    ├── <knowledge_base_A>/     # 每个知识库一个独立文件夹
    └── <knowledge_base_B>/
```

* 同一用户可拥有多个知识库；不同用户的数据物理隔离，互不可见。
* 对同一知识库的多次查询为**增量构建**：新文章追加入库，重复内容自动跳过。
* 用户与知识库通过 `user_id` 与 `kb_name` 标识，由调用方（CLI 参数或上层服务）传入。

## 4. 知识库目录与文件节点架构

每个知识库内部采用规范化的目录结构进行管理，实现正文、资产、数据与元数据的完全解耦。所有文本实体与系统说明统一采用 **Markdown** 格式。

| 路径 | 节点类型 | 核心职能描述与内容规范 |
| --- | --- | --- |
| `/README.md` | 根节点文件 | 提供知识库的全局总览。包含由 AI 生成的关于该主题的执行摘要、文献收集的关键词范围、数据采集的时间戳、以及系统的入口导引说明。 |
| `/index.html` | 根节点文件 | 知识库的全局图形化入口，提供 TAG 分类入口及每个文件的 Meta Data 入口。整合离线全局搜索框控件、可视化动态目录树（DOM 树结构），作为用户双击打开的主界面。 |
| `/Articles/` | 层次化目录 | 承载知识库的核心文本实体。内部包含按子主题（Sub-topics）进一步划分的文件夹，**全量存放由 AI 整理生成的纯 Markdown (.md) 文档**。 |
| `/Attachments/` | 扁平化目录 | 统一剥离并集中存放全库所有的图片、PDF 报告、音频文件等多媒体下载资产，实施极其严格的物理隔离以防止文档夹杂导致性能衰退。 |
| `/Data/` | 隔离型目录 | 存放搜索过程中抓取的结构化数据集（如 CSV, Excel, JSON）。强制下设 `/Raw` 与 `/Processed` 子目录，以区分原始下载文件与 AI 清洗后的数据。 |
| `/_Metadata/` | 系统隐藏目录 | 存放完全剥离于正文内容的、供下游 AI 模型或 RAG（检索增强生成）系统消费的侧边车（Sidecar）JSON 格式元数据标签体系，包含 TAG、分类、来源、时间戳、去重 Hash 及 **AI 生成的文章摘要（Summary）**。 |

## 5. 关键技术要求与规范

### 5.1 统一的文本格式标准

* `/Articles/` 目录下的所有文章必须严格输出为 `.md` 格式，不再混合使用 HTML。
* Markdown 文档内若需引用图片或附件，必须使用相对路径指向 `/Attachments/` 目录（例如 `../Attachments/image.png`）。

### 5.2 元数据（Metadata）与去重机制

* 每个下载或生成的 `.md` 文档，必须在 `/_Metadata/` 目录下拥有路径镜像、同名的 `.json` 侧边车文件（例如 `Articles/AI/foo.md` ↔ `_Metadata/AI/foo.json`）。
* 元数据结构必须包含但不限于：`url`（来源）、`title`、`content_hash`（MD5）、`crawl_time`、`tags`、`category`、`summary`（**LLM 自动生成的文章摘要**）、`source_keywords`（命中该文章的搜索关键词）。
* **去重逻辑**：新数据下载前，优先比对 `/_Metadata/` 中的 URL 或内容 Hash，若已存在则跳过下载或进行版本更新覆盖。

### 5.3 离线交互能力 (`/index.html`)

* **完全离线可用**：所有前端静态资源（JS/CSS）需本地化，不得依赖在线 CDN。
* **动态目录树**：自动读取 `/Articles/` 目录结构与 `.md` 文件列表，渲染为可折叠的 DOM 树。
* **离线搜索与预览**：基于本地 JSON 元数据索引实现轻量级检索，并集成前端 Markdown 渲染器，以便用户直接在 `/index.html` 内预览 Markdown 文章。

### 5.4 技术栈约束

* **Agent 框架**：使用 `deepagents` 构建可交互的研究型 Agent（支持自主规划、多步搜索）。
* **流水线编排**：使用 `langgraph` 的 `StateGraph` 实现确定性的采集流水线（输入优化 → 关键词生成 → 搜索 → 去重 → 摘要 → 入库 → 索引重建）。
* **LLM 与工具抽象**：使用 `langchain` 的模型与工具接口，LLM 与搜索后端均可注入替换（便于测试与更换供应商）。
* **包管理**：使用 `uv` 管理依赖与运行测试。
* **质量要求**：所有新代码必须配有单元测试，单元测试不依赖网络与真实 API Key。
